# 项目答辩常见质疑点解释稿

本文件用于集中整理老师在答辩中可能提出的几个高频问题，并分别给出口头回答版和书面表达版。口头版适合直接在答辩现场使用，书面版适合吸收到报告、README 或附录中。

## 1. 关于 `requirements.txt` 里 `torch==2.10.0` 看起来可疑

### 事实基础

- 当前仓库中的 `requirements.txt` 确实写的是 `torch==2.10.0`
- 当前实际运行环境中，`torch.__version__` 的输出也是 `2.10.0+cu130`
- 项目已经在这个环境中完成训练、调参、评估和 PPT 生成

### 口头回答版

可以直接这样回答：

“这个版本号不是我凭记忆写的，也不是手误，而是我当前实际实验环境中真实安装并验证通过的版本。换句话说，我是先把实验在本机 GPU 环境里跑通，再把实际运行环境同步写进依赖文件，而不是反过来凭常见版本经验去猜。”

如果老师继续追问，可以补一句：

“从课程答辩的角度，真正关键的是依赖文件和实际运行环境是否一致。我这里这一点是自洽的，因为训练、调参、评估的整条链路都已经在这个环境上验证通过。”

### 书面表达版

“本项目中的依赖版本依据实验时的实际可运行环境记录。`torch==2.10.0+cu130` 已在本机 GPU 环境中完成训练、评估与调参验证，因此依赖记录遵循‘与真实运行环境一致’的原则，而非按常见课程环境经验估计填写。”

### 不建议这样说

- “老师记错了”
- “这个版本肯定没问题，不用解释”
- “我也不太清楚，反正能跑”

### 更稳的补强说法

如果老师对版本号敏感，可以再补一句：

“如果需要，我可以现场给出环境验证命令，实际输出就是这个版本号。”

## 2. 关于 `clean` 视图下 `ResNet18` 准确率达到 100%

### 事实基础

- `clean_resnet18` 单次测试结果为：
  - Accuracy `100.00%`
  - Macro F1 `100.00%`
- 基于 3 个不同随机种子分别训练并验证：
  - seed=42：`100.00%`
  - seed=123：`99.79%`
  - seed=2026：`100.00%`
- 平均 Accuracy 为 `99.93%`
- Accuracy 标准差约为 `0.10%`

### 口头回答版

推荐这样答：

“我同意单次 100% 会让人警惕，所以我没有只保留这个数字，而是专门补做了多随机种子稳定性验证。三次 seed 的结果分别是 `100.00% / 99.79% / 100.00%`，平均 Accuracy 是 `99.93%`，标准差只有 `0.10%`。这说明这个高分不是单次偶然，但我也不会把它夸张成任何外部数据都能 100%。”

如果老师进一步追问为什么会这么高，可以继续说：

“我认为这里主要是三点叠加。第一，clean 视图已经做了哈希去重，尽量降低了数据泄漏风险；第二，这个数据集本身类别区分度比较高，病斑纹理和形态差异比较明显；第三，ResNet18 的模型容量和当前 clean 数据规模比较匹配，所以在这个任务上表现非常强。”

### 书面表达版

“`clean_resnet18` 在当前 clean 划分测试集上达到 100% 准确率。考虑到该结果较高，本项目基于 3 个不同随机种子分别训练并验证，结果为 `100.00% / 99.79% / 100.00%`，平均 Accuracy 为 `99.93%`。这表明该高分结果具有较好的稳定性，但并不意味着模型在所有外部数据上都能达到绝对 100%。”

### 更稳的补充表达

“因此，更准确的结论不是‘模型绝对无误’，而是‘在当前 clean 划分和当前任务难度下，模型取得了稳定接近满分的高表现’。”

### 不建议这样说

- “因为模型很强，所以 100% 很正常”
- “既然去重了，就绝对不可能过拟合”
- “这个结果说明模型已经完全学会了”

## 4. 关于 `.gitignore` 排除了 `outputs/`

### 事实基础

- 当前 `.gitignore` 确实排除了 `outputs/`
- 但仓库中已经保留了 `final_assets/` 作为轻量结果目录
- `final_assets/` 中包含：
  - 主结果表
  - 调参结果
  - 稳定性分析
  - 关键图表
  - 实际答辩 PPT 文件

### 口头回答版

推荐这样说：

“这是有意的仓库管理选择，不是遗漏。`outputs/` 里保存的是完整中间产物和模型权重，体积大、数量多，而且很多文件是可以根据代码和配置重新生成的，所以我没有把它们全部纳入 Git。为了兼顾复现和仓库可读性，我单独导出了 `final_assets/`，这里保留了答辩和报告直接需要的轻量结果，包括主结果表、调参结果、关键图表和稳定性分析。”

如果老师继续问 GitHub 能不能复现，可以补一句：

“如果老师从 GitHub 克隆项目，代码、配置、文档和轻量结果都是齐的；如果要完整复现实验过程，只需要按 README 和环境文件重新训练，就能重新生成完整的 `outputs/`。”

### 书面表达版

“`outputs/` 作为完整中间产物目录，包含训练日志、模型权重与评估细节，文件体积较大且可由脚本再生，因此未纳入 Git 跟踪。为保证答辩展示与结果复核，本项目额外提供 `final_assets/` 作为轻量结果目录，保留关键图表、指标汇总和调参结果。”

### 这条解释的核心逻辑

这里一定要强调“复现分两层”：

1. GitHub 仓库层：
   - 查看代码、配置、文档和轻量结果
2. 实验重建层：
   - 按环境文件和脚本重新生成完整 `outputs/`

### 不建议这样说

- “因为太大所以没办法”
- “老师如果要复现就自己跑”
- “中间结果不重要”

## 5. 推荐你在答辩时主动补的一句总括话术

如果老师把这几个问题连续抛出来，你可以用下面这句先稳住全局：

“这几个问题里，有的是沟通层面的风险，比如版本号看起来比较新；有的是实验严谨性要求，比如 clean 下 100% 需要稳定性验证；也有的是工程取舍问题，比如中间产物不全部进 Git。我的处理方式是：对事实不回避、对边界不夸大、对取舍给出理由，并尽量用补充实验和轻量结果目录把答辩风险降下来。”

## 6. 最后给你的答辩默认口径

- `PyTorch 版本问题`
  - 口径：版本号来自真实运行环境，不是手误；关键是依赖文件与实际环境一致
- `100% 准确率问题`
  - 口径：这是 clean 下的稳定高分，而不是“任何情况下都绝对 100%”
- `加分项缺失`
  - 口径：属于增强项尚未全部展开，不影响当前核心结论成立
- `outputs 未入 Git`
  - 口径：属于仓库分层管理，`final_assets` 负责展示，`outputs` 负责可再生产物
